지금 열려 있는 프로젝트(워크스페이스가 아니라 **현재 디렉토리 기준**)를 대상으로 개인정보·보안 스캔을 수행합니다. 어떤 프로젝트에서 실행해도 동작해야 합니다 — 이 워크스페이스 전용 커맨드가 아닙니다.

## 1. 스캔 범위 정하기

- git 저장소면 `git ls-files`로 추적 중인 파일만 대상 (node_modules, .git, dist, build 등은 자동 제외)
- git 저장소가 아니면 현재 디렉토리를 재귀 탐색 (동일하게 대용량 빌드 산출물 폴더는 제외)
- `$ARGUMENTS`에 `history`가 포함되면 작업 트리뿐 아니라 `git log -p`로 커밋 히스토리 전체도 검사 (Public 저장소로 전환하기 전 필수 — 파일을 지워도 과거 커밋에는 남아있음)

## 2. 패턴 스캔

**개인정보 (PII)**
- 주민등록번호 패턴 (`\d{6}[-\s]?[1-4]\d{6}`)
- 전화번호 (`01[0-9][-\s]?\d{3,4}[-\s]?\d{4}`)
- 이메일 주소
- 계좌번호/카드번호 패턴 (14~16자리 숫자, 하이픈 포함/미포함)
- 여권번호, 주소 키워드(시/구/동 + 상세주소 조합)

**시크릿/자격증명**
- API 키 패턴: `sk-`, `AIza`, `ghp_`/`gho_`, `xox[baprs]-`, `AKIA`, `Bearer ey...`
- `-----BEGIN ... PRIVATE KEY-----` 블록
- `.env`, `credentials.json`, `service_account.json` 등이 실수로 git에 추적되고 있는지 (`git ls-files`에 걸리는지)
- 코드 내 하드코딩된 `password =`, `secret =`, `api_key =` 형태의 실제 값

**제3자 개인정보 (가장 놓치기 쉬움)**
- 실명 + 민감 맥락(급여, 계약, 주민번호, 면접평가, 인사정보)이 함께 나오는 문단 — 정규식으로 못 잡으므로, 해당되는 파일을 읽어서 직접 판단
- 원본 보존/백업 폴더(예: `_originals`, `_backup` 등)에 제3자 정보가 실수로 섞여 들어가는 경우가 흔하므로 반드시 검사 대상에 포함

## 3. 결과 보고

심각도별로 그룹화해서 보고:
- **Critical**: 시크릿/자격증명, 주민등록번호, 제3자 민감정보
- **High**: 전화번호/이메일/주소 등 본인 PII
- **Review 필요**: 정규식으로는 확신할 수 없어 사람 판단이 필요한 항목

각 항목마다: 파일 경로, 몇 번째 줄/커밋인지, 왜 문제인지, 제안하는 조치(redact / 파일 삭제 / `.gitignore` 추가 / 자격증명 로테이션)를 함께 제시.

## 4. 조치는 허락받고 진행 (반자동화)

**절대 스캔 결과만 보고 자동으로 수정·삭제하지 않는다.** 보고 후 사용자에게 "이 항목들 조치할까요?"라고 명확히 물어보고, 승인받은 항목만 Edit/Bash로 실제 조치를 진행한다. 항목이 여러 개면 일괄 승인도 가능하지만, Critical 등급(시크릿·제3자 정보)은 개별 확인을 권장한다.

커밋 히스토리에 이미 노출된 항목은 파일만 지워서는 해결되지 않는다는 점을 반드시 명시하고, 필요하면 `git filter-repo` 등 히스토리 재작성이 필요하다는 것도 안내만 하고 실행은 승인 후에만 한다 (히스토리 재작성은 파괴적 작업이라 특히 신중하게).
